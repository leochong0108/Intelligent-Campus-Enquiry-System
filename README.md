A Retrieval-Augmented Generation (RAG) chatbot designed to automate university inquiries, reducing administrative workload by providing accurate, verifiable answers 24/7.

📖 Overview

The Smart Information Counter solves the problem of limited availability and "hallucinations" in traditional AI chatbots. Unlike generic LLMs, this system retrieves answers strictly from a verified university Knowledge Base (KB).

It features a Hybrid Search Engine (Vector Semantic Search + Fuzzy Logic) to ensure high accuracy. If the AI cannot find an answer with high confidence (>65%), it gracefully falls back to a "Request Human Assistance" flow, creating a Human-in-the-Loop feedback cycle where administrators can convert failed queries into new training data.
<br><br>

✨ Key Features

🧠 AI & Search Logic

1. RAG Architecture: utilizing Google Gemini Pro for natural language generation based only on retrieved context.

2. Custom Vector Search: Implemented Cosine Similarity algorithms natively in PHP to match user queries with FAQ embeddings.

3. Hybrid Fallback: Uses Fuzzy Search (Levenshtein distance) if semantic vector search fails, ensuring robustness against typos.

4. Automated Embeddings: New FAQs are automatically converted to vector embeddings upon creation via Model Observers.

🛠️ Technical Implementation Highlights

The Search Algorithm

Rather than relying on an external Vector Database (like Pinecone), I implemented a lightweight vector comparison logic within the application service layer to keep infrastructure costs low.
<br><br>
// Creating a lightweight Cosine Similarity check in PHP

```
private function cosineSimilarity(array $vecA, array $vecB)
{
    $dotProduct = 0;
    $magnitudeA = 0;
    $magnitudeB = 0;

    foreach ($vecA as $key => $value) {
        if (!isset($vecB[$key])) continue;
        $dotProduct += $value * $vecB[$key];
        $magnitudeA += $value ** 2;
        $magnitudeB += $vecB[$key] ** 2;
    }

    return ($magnitudeA * $magnitudeB) == 0 ? 0 : $dotProduct / (sqrt($magnitudeA) * sqrt($magnitudeB));
}
```

💡 Challenges & Learnings

1. Handling Hallucinations: Generic LLMs often invent facts. I solved this by enforcing a strict similarity threshold (0.65). If the vector match is below this, the system refuses to answer, preventing misinformation.

2. Latency Optimization: Generating embeddings and getting AI responses can be slow. I optimized this by indexing the database and running the "Reasoning" phase separately from the "Generation" phase.

3. Data Consistency: Used Database Transactions (DB::transaction) when converting logs to FAQs to ensure no data is lost if the API connection drops mid-process.
