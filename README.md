A Retrieval-Augmented Generation (RAG) chatbot designed to automate university inquiries, reducing administrative workload by providing accurate, verifiable answers 24/7.
📖 Overview
The Smart Information Counter solves the problem of limited availability and "hallucinations" in traditional AI chatbots. Unlike generic LLMs, this system retrieves answers strictly from a verified university Knowledge Base (KB).
It features a Hybrid Search Engine (Vector Semantic Search + Fuzzy Logic) to ensure high accuracy. If the AI cannot find an answer with high confidence (>65%), it gracefully falls back to a "Request Human Assistance" flow, creating a Human-in-the-Loop feedback cycle where administrators can convert failed queries into new training data.

✨ Key Features
🧠 AI & Search Logic
RAG Architecture: utilizing Google Gemini Pro for natural language generation based only on retrieved context.
Custom Vector Search: Implemented Cosine Similarity algorithms natively in PHP to match user queries with FAQ embeddings.
Hybrid Fallback: Uses Fuzzy Search (Levenshtein distance) if semantic vector search fails, ensuring robustness against typos.
Automated Embeddings: New FAQs are automatically converted to vector embeddings upon creation via Model Observers.

🖥️ User Interface (Visitor)
Voice Input: Integrated Web Speech API for accessibility.
Real-time Chat: Responsive Vue.js SPA interface.
Idle Management: Automated session timeout for public kiosk security.

⚙️ Admin Dashboard
Analytics Visualization: Chart.js integration to track query trends, popular intents, and department loads.
AI-Generated Reports: One-click executive summaries generated by Gemini based on raw log data.
Knowledge Management: Batch import (Excel/CSV) for FAQs and Departments.
Fail Log Recovery: Administrators can review failed queries and convert them into FAQs with one click, closing the knowledge gap.

🛠️ Technical Implementation Highlights
The Search Algorithm
Rather than relying on an external Vector Database (like Pinecone), I implemented a lightweight vector comparison logic within the application service layer to keep infrastructure costs low.

// Creating a lightweight Cosine Similarity check in PHP
private function cosineSimilarity(array $vecA, array $vecB)
{
    $dotProduct = 0;
    $magnitudeA = 0;
    $magnitudeB = 0;

    foreach ($vecA as $key => $value) {
        if (!isset($vecB[$key])) continue;
        $dotProduct += $value * $vecB[$key];
        $magnitudeA += $value ** 2;
        $magnitudeB += $vecB[$key] ** 2;
    }

    return ($magnitudeA * $magnitudeB) == 0 ? 0 : $dotProduct / (sqrt($magnitudeA) * sqrt($magnitudeB));
}

💡 Challenges & Learnings
Handling Hallucinations: Generic LLMs often invent facts. I solved this by enforcing a strict similarity threshold (0.65). If the vector match is below this, the system refuses to answer, preventing misinformation.
Latency Optimization: Generating embeddings and getting AI responses can be slow. I optimized this by indexing the database and running the "Reasoning" phase separately from the "Generation" phase.
Data Consistency: Used Database Transactions (DB::transaction) when converting logs to FAQs to ensure no data is lost if the API connection drops mid-process.
